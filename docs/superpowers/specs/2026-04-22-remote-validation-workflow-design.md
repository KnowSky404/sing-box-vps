# 远程验证工作流设计

- 日期：2026-04-22
- 适用范围：本地开发工作流、远程验证调度、测试机执行剧本、验证产物归档
- 目标：为 `sing-box-vps` 建立“本地快速回归 + 按改动范围触发的远程真实验证”工作流，在功能新增或修复后自动校验安装交互、`systemd` 启动和运行态问题

## 1. 背景

当前仓库已经有一批 `tests/*.sh`，它们擅长验证：

- 文案与菜单输出
- 配置生成逻辑
- 状态迁移和局部回归

但它们主要运行在本地受控环境里，无法覆盖真实 VPS 上的关键运行态问题，例如：

- `systemd` service 是否真正安装、启动和重启成功
- 真实文件路径、权限和残留状态是否符合预期
- 端口监听、日志输出和运行时健康状态是否正常
- 从“首次安装 -> 修改配置 -> 查看状态 -> 卸载/重装”的完整生命周期是否稳定

纯 Docker 对这类问题有帮助，但对本项目不够充分，因为普通容器无法完整模拟 `systemd`、宿主机级网络与系统服务行为。项目已经有一台可污染、可重装的固定 VPS 可用，因此更适合将远程测试机纳入正式工作流。

## 2. 设计目标

- 保留现有本地脚本测试作为快速回归层
- 基于本次改动文件范围，自动决定是否触发远程验证
- 当命中核心运行路径时，自动通过 SSH 在固定测试 VPS 上执行标准化验证剧本
- 覆盖首次安装、已有安装修改、卸载重装和运行态冒烟检查
- 对每轮远程验证统一采集日志、配置和状态产物，便于定位失败原因
- 失败时优先保留现场，不做隐式恢复或自动 DD

## 3. 非目标

- 本轮不实现自动 DD、自动重装系统或自动快照回滚
- 本轮不要求把所有现有测试全部迁移到 Docker 或远程机
- 本轮不构建复杂的测试编排平台，只做项目内可维护的脚本化工作流
- 本轮不把固定测试 VPS 当作长期干净环境；只要求它是可污染、可清理、必要时可重装的验证靶机

## 4. 总体方案

工作流分为三层：

1. 本地脚本级回归层
   - 继续运行仓库内现有 `tests/*.sh`
   - 用于快速发现回归，作为最便宜的第一道防线
2. 本地集成预检层
   - 负责轻量验证，例如配置生成、`sing-box check`、菜单流程中的非系统级逻辑
   - 是否引入 Docker 不是本设计的阻塞项；后续可增量接入
3. 远程真实验证层
   - 当改动命中核心运行路径时，自动 SSH 到固定测试 VPS
   - 执行标准场景剧本，验证真实安装、`systemd`、端口监听和运行态

设计原则：

- 慢测试只在高风险改动上触发
- 远程验证必须可重复、可采集、可解释
- 失败优先保留现场，方便排障
- 工作流应兼容未来增加“验证后自动 DD”的扩展能力

## 5. 触发规则

### 5.1 触发方式

基于本次改动文件列表判断是否触发远程验证，而不是每次改动都固定执行远程剧本。

### 5.2 命中远程验证的路径

当本次改动命中以下文件或目录时，自动触发远程验证：

- `install.sh`
- `uninstall.sh`
- `utils/`
- `configs/`

### 5.3 默认不触发远程验证的路径

仅改动以下内容时，不自动触发远程验证：

- `README.md`
- `docs/`
- `LICENSE`
- 纯注释、纯文案、纯格式调整
- 仅 `tests/`

说明：

- `tests/` 的改动默认不反向触发远程验证，避免“只改测试也跑昂贵的远程流程”
- 若一次提交同时改动了核心运行路径与上述跳过路径，仍然触发远程验证

## 6. 验证流水线

调度逻辑固定为：

1. 识别本次改动文件列表
2. 运行本地快速测试
3. 若命中远程触发规则，则自动执行远程验证
4. 收集远程验证产物并汇总结果
5. 任一关键步骤失败则终止，并输出失败原因与产物目录

行为约束：

- 本地快速测试失败时，不进入远程验证
- 远程验证执行前必须输出目标主机信息，防止误连非测试机
- 远程验证期间必须持有测试机占用锁，避免并发污染

远程验证中的运行态检查分为两层：

1. 服务端运行态检查
   - 继续验证 `systemd`、监听端口、配置文件与日志
2. 协议级闭环检查
   - 在测试机本机额外启动一个临时客户端
   - 让客户端连接测试机自身的服务端入站
   - 再通过客户端代理访问测试机本机的临时 HTTP 探针服务

这样可以把“服务已经启动”和“协议实际可用”区分开，避免仅凭 `sing-box check`、`systemctl` 和监听端口误判通过。

## 7. 远程测试机策略

测试机是一台长期存在的固定 VPS，但在工作流语义上将其视为：

- 可污染
- 可清理
- 必要时可重装

本轮策略：

- 每次远程验证前先检查机器是否存在旧安装残留
- 若执行场景要求干净基线，则先走标准清理流程
- 失败后默认保留现场，不自动恢复

后续可以在不改变本设计主体的前提下，将“标准清理流程”升级为“自动 DD 重装后再执行验证”。

## 8. 远程验证场景

第一版仅保留 4 个标准远程场景。

### 8.1 `fresh_install_vless`

用途：

- 验证清理后的机器是否能完成首次安装
- 覆盖安装交互、配置落盘、service 创建和启动

### 8.2 `reconfigure_existing_install`

用途：

- 验证已有安装基础上的修改流程
- 覆盖配置变更、状态同步、服务重启和节点信息刷新

### 8.3 `uninstall_and_reinstall`

用途：

- 验证卸载清理、重装与幂等性
- 覆盖残留处理和二次安装稳定性

### 8.4 `runtime_smoke`

用途：

- 对当前运行态做快速远程确认
- 不重新安装，只验证服务、配置、端口和日志是否正常

## 9. 场景与改动类型映射

默认映射规则如下：

- 改动 `install.sh`、`uninstall.sh`、`utils/`、`configs/`
  - 执行 `fresh_install_vless`
  - 执行 `reconfigure_existing_install`
  - 执行 `runtime_smoke`
- 若改动明确涉及卸载、接管、幂等、残留恢复
  - 额外执行 `uninstall_and_reinstall`
- 仅改动 `tests/`
  - 默认只跑本地测试，不强制远程验证

这套映射优先保证“高风险核心改动一定覆盖真实场景”，同时避免固定 VPS 上的无意义全量执行。

## 9.1 协议级闭环检查策略

远程场景完成安装、重配或运行态确认后，必须追加一次协议级闭环检查。

闭环检查固定流程：

1. 在测试机本机启动一个临时 HTTP 探针服务
2. 读取 `/root/sing-box-vps/protocols/index.env`，识别全部已安装协议
3. 为每个已安装协议尝试生成对应的临时客户端配置
4. 启动临时客户端，通过其本地代理去访问临时 HTTP 探针服务
5. 记录每个协议的探测结果、日志和请求输出

行为约束：

- 必须逐个检查所有已安装协议，而不是只检查本场景的主协议
- 第一批优先实现 `vless-reality` 和 `hy2`
- 尚未实现客户端生成器的协议，结果记为 `unsupported`
- `unsupported` 不直接导致场景失败，但必须在产物和汇总中明确暴露
- 任一“已支持协议”的闭环探测失败，则当前远程场景失败

选择本机临时 HTTP 探针服务而不是外部网站的原因：

- 不依赖公网连通性和外部站点可用性
- 更容易保证回归结果稳定
- 可以把失败收敛为“客户端配置、协议握手、服务端入站、代理转发”中的一环

## 10. 每个远程场景的检查项

### 10.1 `fresh_install_vless`

必须检查：

- 基线清理后不存在旧的 `sing-box` service、`/root/sing-box-vps`、`/usr/local/bin/sbv`
- 通过非交互或受控交互方式完成一次 VLESS + REALITY 安装
- 关键文件存在：
  - `/root/sing-box-vps/config.json`
  - `/root/sing-box-vps/protocols/index.env`
  - `/usr/local/bin/sbv`
  - `/etc/systemd/system/sing-box.service`
- `sing-box check -c /root/sing-box-vps/config.json` 成功
- `systemctl is-active sing-box` 为 `active`
- 目标端口处于监听状态
- `journalctl -u sing-box -n 100 --no-pager` 中没有明显启动失败关键字
- `sbv` 的状态/节点信息路径输出正常
- 对全部已安装协议执行一次协议级闭环检查

### 10.2 `reconfigure_existing_install`

必须检查：

- 在已有安装基础上执行一次受控修改，例如端口变更或协议配置更新
- 修改后状态文件与 `config.json` 同步变化
- `systemctl restart sing-box` 后服务仍为 `active`
- 新端口监听正常，旧端口不再监听
- 关键配置的修改前后 diff 被保存为产物
- 再次执行状态/节点信息查看，确认输出与新配置一致
- 对全部已安装协议执行一次协议级闭环检查，确认修改后客户端仍可连通

### 10.3 `uninstall_and_reinstall`

必须检查：

- 执行卸载脚本或菜单卸载路径
- service、二进制、工作目录和命令入口按预期清理
- 在卸载后的机器上重新完成安装
- 再次执行 `sing-box check`、`systemctl is-active` 和端口监听检查
- 重装结果不受旧状态污染
- 对重装后全部已安装协议执行一次协议级闭环检查

### 10.4 `runtime_smoke`

必须检查：

- `systemctl is-active sing-box`
- `systemctl status sing-box --no-pager`
- `journalctl -u sing-box -n 100 --no-pager`
- `sing-box check -c /root/sing-box-vps/config.json`
- 当前监听端口结果
- 核心文件存在性
- 关键目录大小与最近修改时间快照
- 对全部已安装协议执行一次协议级闭环检查，但不重装服务端

## 10.5 协议级闭环检查项

每次协议级闭环检查至少包含以下步骤：

- 启动测试机本机临时 HTTP 探针服务，例如返回固定内容 `sing-box-vps-loopback-ok`
- 为当前协议生成临时客户端配置，客户端出站必须连接测试机本机的服务端入站
- 启动临时客户端，并开放一个临时本地代理监听端口
- 通过 `curl` 或等价方式，经由临时客户端代理访问本机 HTTP 探针服务
- 校验响应正文包含预期探针内容
- 停止临时客户端和临时 HTTP 探针服务

第一批协议生成器目标：

- `vless-reality`
- `hy2`

结果语义固定为：

- `success`：该协议已实现闭环探测，且探针请求成功
- `failure`：该协议已实现闭环探测，但客户端启动、握手、代理请求或内容校验失败
- `unsupported`：该协议当前未实现客户端生成器，因此未执行探测

## 11. 产物归档

每轮远程验证都必须在本地生成一份结果目录，例如：

`dev/verification-runs/<timestamp>/`

目录内至少保存：

- 本次改动文件列表
- 调度汇总信息（例如 `mode`、远程是否跳过、最终状态）
- 实际触发的远程场景列表
- 每个场景的 stdout/stderr
- `systemctl status` 输出
- `journalctl` 输出
- 远程 `config.json` 副本
- 监听端口检查结果
- 关键 diff 或快照
- 每个协议的闭环探测结果
- 每个协议的临时客户端配置副本或脱敏快照
- 每个协议的客户端运行日志
- 每个协议的探针请求输出
- 最终汇总状态

产物要求：

- 路径稳定，便于人工排查和后续脚本消费
- 场景失败时必须保留
- 即使成功也应至少保留最近若干轮，方便回归对比
- 协议级闭环探测必须按协议单独分目录，避免多协议场景下日志混淆

## 12. 失败处理策略

### 12.1 本地测试失败

- 直接终止
- 不进入远程验证

### 12.2 SSH 连通性失败

- 直接终止
- 将 SSH 错误写入本地产物目录

### 12.3 远程场景失败

- 立即停止后续场景
- 默认不自动清理现场
- 优先保留测试机当前状态供排查

补充约束：

- 若失败来自某个已支持协议的闭环探测，必须保留该协议的客户端配置、日志和探针请求输出
- 若仅出现 `unsupported` 协议，不应把场景误判为成功覆盖全部协议

### 12.4 日志异常

即使命令返回成功，只要日志中出现明确的启动失败或严重错误关键字，也视为场景失败。

### 12.5 禁止的默认行为

第一版明确不做：

- 失败后自动 DD
- 失败后自动重装系统
- 失败后自动重试覆盖原现场

原因：

- 自动恢复会破坏首次失败现场，降低问题定位效率
- 自动 DD 更适合作为后续单独 skill 的显式动作，而不是默认隐藏行为

## 13. 实现边界与结构建议

建议新增一个本地“验证调度脚本”，负责：

- 识别改动文件列表
- 支持通过 `--changed-file <path>...` 注入显式改动集，便于本地回归测试
- 判断是否触发远程验证
- 执行本地快速测试
- 在 `VERIFY_SKIP_REMOTE=1` 时仅完成本地调度与产物写入，不占用测试机
- 申请测试机锁
- SSH 到固定测试机执行标准场景剧本
- 拉回产物并输出汇总结果

建议将远程执行内容做成仓库内固定脚本，而不是在本地通过内联命令临时拼接。原因：

- 更易维护
- 更易复用
- 更易审计
- 后续接入自动 DD 时更容易替换环境初始化阶段

## 14. 风险与控制

- 风险：远程验证过于频繁，拖慢开发速度
  - 控制：仅对核心运行路径触发远程验证
- 风险：固定测试机存在历史污染，导致结果不稳定
  - 控制：对需要干净基线的场景先执行标准清理流程
- 风险：误连非测试机
  - 控制：远程验证前强制输出目标主机标识，并设置测试机占用锁
- 风险：失败后现场被覆盖，难以排查
  - 控制：第一版默认失败即停，不自动恢复

## 15. 验收标准

本设计落地后，应满足：

1. 核心脚本改动会自动触发本地测试与远程真实验证
2. 非核心改动不会无意义地占用固定测试机，且可用 `--changed-file` 为调度逻辑补充回归测试
3. 远程验证至少覆盖首次安装、已有安装修改和运行态检查
4. 远程验证中的已安装协议会逐个执行客户端到服务端的闭环探测
5. 暂未支持闭环探测的协议会被明确标记为 `unsupported`，而不是静默跳过
6. 远程失败时可以直接从本地产物定位失败阶段、具体协议和关键日志
7. 后续接入自动 DD 时无需推翻当前触发规则、场景模型和产物结构
