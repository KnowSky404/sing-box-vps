# 安装前现有实例探测与接管设计

- 日期：2026-04-21
- 适用范围：`install.sh` 安装入口、状态恢复逻辑、运行时补齐逻辑、相关测试
- 目标：在安装入口统一探测现有 `sing-box` 实例状态，将现场区分为“全新 / 完整 / 残缺”，并在残缺场景下允许用户选择“接管现有实例”或“按全新安装处理”

## 1. 背景

当前 `install.sh` 判断“是否已安装”的方式过于粗糙，主要依赖：

- `SINGBOX_BIN_PATH` 是否存在
- `SINGBOX_CONFIG_FILE` 是否存在

这在理想路径下够用，但对真实 VPS 运维和测试场景不够稳健。常见残缺现场包括：

- `config.json` 还在，但二进制被删了
- systemd service 丢了，但配置目录还在
- `index.env` 或协议状态文件丢失，导致菜单无法正确识别已安装协议
- 旧版单协议配置存在，但状态层尚未迁移
- `sbv` 全局命令丢失，但运行实例仍然存在

这类现场目前既不算真正的“全新安装”，也不总能进入稳定的“更新/追加协议”流程，导致用户必须手动排查或先卸载再重装。

## 2. 设计目标

- 为安装入口增加统一的现有实例探测层，而不是散落在各分支里各自判断
- 将实例现场归一为 `fresh`、`healthy`、`incomplete` 三种状态
- 当状态为 `incomplete` 时，让用户显式选择：
  - 接管现有实例
  - 按全新安装处理
- 接管现有实例时：
  - 以 `config.json` 为真源恢复协议状态层
  - 补齐缺失的二进制、systemd service 和 `sbv`
  - 校验配置是否可被 `sing-box` 正常加载
- 保持现有“首次安装”“更新二进制”“新增协议”主流程不被重写，只在前面增加探测与分流

## 3. 非目标

- 不自动删除残缺现场
- 不在本轮实现“按全新安装处理”时的自动清理或自动迁移
- 不支持任意手工魔改目录结构；仍以当前项目约定路径为前提
- 不尝试从无效 JSON 或完全未知入站结构中推断协议状态

## 4. 统一探测模型

### 4.1 探测对象

安装入口统一探测以下对象：

- `SINGBOX_BIN_PATH`
- `SINGBOX_SERVICE_FILE`
- `SINGBOX_CONFIG_FILE`
- `SB_PROTOCOL_INDEX_FILE`
- `SB_PROTOCOL_STATE_DIR` 下协议状态文件
- `SBV_BIN_PATH`

### 4.2 探测结果

探测结果归一为三类：

- `fresh`
  - 没有发现足以表明现有实例存在的关键对象
  - 直接进入首次安装流
- `healthy`
  - 存在可运行实例，且状态层与运行时关键对象基本完整
  - 直接进入现有“更新 sing-box / 安装新增协议”流程
- `incomplete`
  - 发现实例痕迹，但运行时组件、状态层或入口组件存在缺失或不一致
  - 进入“接管 / 全新安装处理”分流菜单

### 4.3 判定原则

- `config.json` 是“是否存在实例痕迹”的最高优先级依据
- `index.env` 和协议状态文件属于可重建状态缓存，不作为最终真源
- `sbv` 丢失不表示实例不存在，但属于残缺现场的一部分
- `service` 或二进制缺失时，只要配置真源还在，也应判为 `incomplete` 而不是 `fresh`

## 5. 安装入口分流

`install_or_update_singbox()` 不再直接用“二进制 + 配置文件”判断已安装，而是先调用统一探测入口。

### 5.1 `fresh`

- 直接进入现有 `install_or_reconfigure_singbox`

### 5.2 `healthy`

- 加载当前状态
- 进入现有“更新二进制 / 安装新增协议”菜单

### 5.3 `incomplete`

展示新的异常实例处理菜单：

1. 接管现有实例
2. 按全新安装处理
0. 返回

行为约束：

- 不自动接管
- 不自动拒绝
- 必须由用户显式选择

## 6. 接管现有实例

### 6.1 状态层恢复

接管时按以下顺序恢复状态层：

1. 执行现有 `migrate_legacy_single_protocol_state_if_needed`
2. 基于 `config.json` 识别当前协议集合
3. 重建或修正 `SB_PROTOCOL_INDEX_FILE`
4. 为缺失的协议状态文件回填对应 `.env` 状态

恢复原则：

- 若 `index.env` 与 `config.json` 冲突，以 `config.json` 为准
- 若状态文件缺失但 `config.json` 可识别，则重建状态文件
- 若 `config.json` 无法识别出受支持的协议，则停止接管

### 6.2 运行时组件补齐

接管时同时检查并补齐：

- `sing-box` 二进制
- systemd service
- `sbv` 全局命令

补齐原则：

- 只补缺，不强制覆盖现有看起来正常的对象
- 二进制缺失时按当前适配版本补装
- service 缺失时复用现有 `setup_service`
- `sbv` 缺失时复用现有全局命令安装逻辑

### 6.3 配置校验

状态层恢复和运行时补齐完成后，必须执行：

```bash
sing-box check -c /root/sing-box-vps/config.json
```

校验规则：

- 成功：接管成功，可进入现有更新/追加协议流程
- 失败：接管中止，提示“当前配置不可接管”，不继续后续管理流程

### 6.4 接管结果摘要

接管成功后，向用户展示简短摘要：

- 恢复了哪些协议状态
- 是否补装了二进制
- 是否重建了 service
- 是否补装了 `sbv`

这样便于测试和排障确认脚本到底做了什么。

## 7. 按全新安装处理

用户在 `incomplete` 状态下选择“按全新安装处理”时：

- 不自动删除旧目录
- 不自动覆盖旧配置
- 明确提示当前残缺现场仍存在，建议先执行卸载
- 允许继续走首次安装流程

这样保持用户控制权，避免脚本在残缺现场上静默做破坏性覆盖。

## 8. 异常与拒绝条件

以下情况不进入自动接管成功路径：

- `config.json` 存在但不是合法 JSON
- `config.json` 中的协议类型无法识别为当前支持集合
- 状态层恢复后 `sing-box check` 失败

这些场景统一处理为：

- 提示发现旧实例痕迹
- 明确说明当前配置不可接管
- 建议用户查看错误或先卸载后重装

## 9. 实现方式

推荐新增一组聚焦 helper，而不是继续把现场判断散落在安装入口：

- 一个实例探测 helper，负责汇总各路径存在性和一致性
- 一个实例接管 helper，负责状态恢复、组件补齐、配置校验和摘要输出
- 一个从 `config.json` 推导协议集合的 helper，供接管与状态恢复共同使用

保持边界：

- `config.json` 是运行真源
- 协议状态层是可重建缓存
- 安装入口只负责分流，不承载全部修复细节

## 10. 测试范围

本轮至少覆盖以下自动化测试：

1. 残缺现场会进入“接管现有实例 / 按全新安装处理”菜单
2. 选择接管时，缺失的 `index.env` 和协议状态文件可从 `config.json` 恢复
3. 选择接管时，缺失的二进制、service 和 `sbv` 会被补齐
4. `config.json` 非法时，接管失败且不会进入更新流程

必要时可补一条回归测试：

- `healthy` 场景仍直接进入原有更新菜单，不受新探测层干扰

## 11. 风险与控制

- 风险：把原本应视为“残缺”的现场误判成 `healthy`
  - 控制：将 `service`、二进制、状态层和 `sbv` 一并纳入统一探测，不再只看两个文件
- 风险：状态恢复与旧迁移逻辑相互覆盖
  - 控制：保持 `migrate_legacy_single_protocol_state_if_needed` 先执行，再做更通用的 reconcile
- 风险：坏配置被补齐 service 后重新拉起
  - 控制：必须先通过 `sing-box check`，失败则中止接管
- 风险：用户误以为“按全新安装处理”等于自动清理旧现场
  - 控制：在菜单与提示文案中明确“不会自动删除旧现场”

## 12. 结论

本轮应在 `install.sh` 安装入口前增加统一实例探测与异常分流层，以 `config.json` 为现有实例真源，将协议状态层视为可重建缓存，并允许用户在残缺现场中手动选择“接管现有实例”或“按全新安装处理”。这样既能覆盖真实运维与测试中的残缺安装问题，也能保持现有安装/更新主流程的稳定性。
