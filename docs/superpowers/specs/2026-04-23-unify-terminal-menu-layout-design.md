# 终端菜单统一左对齐布局设计

- 日期：2026-04-23
- 适用范围：`install.sh` 中所有数字选项菜单
- 目标：将当前仍残留的旧式“全屏居中/悬浮式”菜单统一收敛到主菜单现有的左对齐控制台面板风格，消除菜单系统中的视觉割裂

## 1. 背景

当前仓库已经经历过两轮终端 UI 调整：

- 第一轮引入了宽度感知、分区标题和统一菜单项 helper
- 第二轮把主菜单品牌区改为左对齐，修正了“顶部居中、主体贴左”的割裂感

但菜单系统仍不完全统一，问题主要集中在：

- 一些子菜单和轻量选择页仍沿用旧的居中页头或旧式 `echo` 拼接输出
- 主菜单已经具备明确的左对齐视觉基准，但部分二级菜单仍像“悬在屏幕中央的一块”
- 小型确认菜单、安装流中的协议选择菜单、节点信息下的选择页等分散页面没有完全纳入同一布局体系

用户当前反馈的核心不是单页文案或配色问题，而是整个菜单系统里仍存在“新旧 UI 混用”的残留观感问题。

## 2. 设计目标

- 所有数字选项菜单统一采用左对齐控制台面板风格
- 主菜单与子菜单共享同一视觉起点，不再存在主菜单贴左、子菜单明显居中的两套对齐体系
- 轻量选择菜单也纳入统一规则，避免“小菜单例外”
- 保留当前菜单编号、动作映射、交互方式和主要信息结构
- 将渲染规则收敛到统一 helper，避免后续新增菜单再次回到旧样式

## 3. 非目标

- 不改动菜单编号及其分发逻辑
- 不改动安装、更新、卸载、Warp、协议配置等业务行为
- 不把日志输出、二维码输出、实时日志、完整节点正文信息改成卡片式菜单
- 不重做配色体系或引入额外 TUI 依赖
- 不借本轮菜单统一去做大规模模块拆分

## 4. 推荐方案

采用“统一左对齐页面骨架 + 菜单模板分层”的方案。

- 保留主菜单现有的左对齐品牌块作为最高层视觉基准
- 新增或替换子菜单通用页头 helper，使其不再走旧式居中路径
- 将全部数字选项菜单归入同一套页面骨架和菜单项渲染规则
- 只在信息密度上区分“主菜单 / 标准子菜单 / 轻量选择菜单”，不再区分对齐方式

该方案能一次性解决风格分裂问题，也比“逐页补丁式替换”更利于后续维护。

## 5. 统一后的页面骨架

所有数字菜单页统一遵循以下结构：

1. 左对齐品牌或页面标题区
2. 当前状态摘要区（需要时）
3. 菜单分区标题或单组菜单项
4. 底部输入提示或返回提示

约束如下：

- 主菜单继续保留完整品牌信息
- 子菜单改为左对齐页面头，不再保留明显悬中的标题块
- 小型确认菜单即使只有 2 到 4 个选项，也要复用同一骨架
- 页面之间的差异体现在信息多少，而不是对齐模式不同

## 6. 页面模板分类

### 6.1 主菜单模板

用于 `main`。

特点：

- 顶部保留左对齐品牌块
- 主体保持当前分区结构
- 继续作为整个脚本的唯一品牌入口

### 6.2 标准子菜单模板

用于系统管理、协议栈管理、Warp 管理、流媒体检测、节点信息二级菜单等中等信息量页面。

统一包含：

- 左对齐标题
- 一到两行副标题或说明
- 摘要块
- 菜单项列表
- 底部输入提示

### 6.3 轻量选择菜单模板

用于安装/更新中的协议选择、输出方式选择、接管现有实例、Warp 路由模式、查看链接/二维码/导出等小型菜单。

统一包含：

- 左对齐标题
- 简短上下文说明
- 菜单项列表
- 输入提示

可省略摘要块和分区标题，但不可回退到旧式居中样式。

## 7. 纳入改造的页面范围

本次统一范围是“凡是数字选项菜单”的页面，包括但不限于：

- 主菜单
- 系统管理及其下级菜单
- Warp 管理相关菜单
- 流媒体检测菜单
- 节点信息路径下的各类选择页
- 安装/更新过程中的协议选择、小型确认菜单、输出方式选择菜单
- 接管现有实例、全新安装处理等分流选择菜单
- 其他本质上通过数字编号让用户做选择的页面

## 8. 明确不纳入的页面范围

以下内容保持现有文本流输出方式，只做最小必要的留白或标题兼容，不纳入统一菜单模板：

- 安装过程日志
- 配置校验结果
- 完整节点信息正文
- 二维码展示
- 实时日志或其他流式输出页面

## 9. 实现方式

### 9.1 Helper 收口原则

不再允许菜单函数自行拼接大量 `echo`、空行和对齐细节。

实现上应：

- 保留现有通用菜单项和宽度相关 helper
- 新增或改造统一的左对齐页面头 helper
- 为标准子菜单和轻量选择菜单提供统一渲染入口
- 让各菜单函数只负责声明标题、副标题、摘要内容和菜单项

视觉相关逻辑，包括：

- 左边界
- 内部缩进
- 分区标题样式
- 菜单项对齐
- 摘要块风格
- 输入提示位置

都应尽量集中在 helper 层统一处理。

### 9.2 兼容性要求

- 继续遵守当前终端宽度探测与窄终端回退策略
- 左对齐不等于贴边输出，应保留现有安全边距和可读性
- 颜色能力不足时保留层级结构，不依赖颜色本身表达布局

## 10. 测试与验证

实现完成后至少应覆盖以下验证：

- 主菜单保持现有左对齐品牌风格
- 至少一个标准子菜单改造后不再出现旧式居中页头特征
- 至少一个轻量选择菜单改造后与主菜单视觉基线一致
- 窄终端下菜单仍能稳定回退，不出现换行错位
- 已有主菜单、系统管理菜单相关测试保持通过或同步更新
- 新增回归测试覆盖“标准子菜单模板”和“轻量选择菜单模板”各至少一类页面

## 11. 风险与控制

- 风险：安装流程中的小菜单分散，容易漏改
  - 控制：在实现前先系统性检索所有菜单入口和 `render_menu_item` 使用点
- 风险：不同菜单历史包袱较重，统一时容易继续做页面特例
  - 控制：优先补齐 helper，再迁移页面，避免直接在业务函数里堆样式代码
- 风险：测试覆盖不足导致后续回退到旧 UI
  - 控制：补上模板级回归测试，而不是只靠单页人工观察

## 12. 结论

本次应把“所有数字选项菜单统一成主菜单这套左对齐控制台面板风格”作为明确目标推进。实现重点不是重新设计终端 UI，而是彻底收口已有菜单渲染路径，消除旧式居中菜单残留，并为后续新增菜单建立统一的默认样式基线。
