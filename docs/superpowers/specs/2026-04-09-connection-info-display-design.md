# Connection Info Display Design

**Date:** 2026-04-09

**Goal**

收敛 `sing-box-vps` 在安装、升级、修改配置和手动查看信息时的终端输出，避免在非首次场景下直接刷出长链接和二维码，并让用户按明确选项查看所需的连接信息。

## Problem

当前 `install.sh` 中的 `display_info()` 同时承担了两类职责：

- 展示服务状态、协议、端口、Warp 状态、配置文件路径等运行摘要。
- 直接输出链接、Mixed 代理地址和二维码。

这导致以下问题：

- 首次安装之外的路径，例如版本升级、配置变更、手动查看状态，也会自动打印大量连接信息。
- `vless+reality` 会直接输出完整链接和 ANSI 二维码，信息密度过高。
- `mixed` 会直接输出多条代理地址，但缺少明确的“协议说明 -> 具体链接”结构。
- 用户进入菜单 `8` 时没有细分查看方式，无法只看链接或只看二维码。

## Approved Decisions

### 1. 采用展示拆层方案

采用“状态摘要”和“连接信息展示”分离的方案：

- `display_status_summary()` 只显示摘要信息。
- `show_connection_info_menu()` 只负责菜单 `8` 的交互式连接展示。
- `show_post_config_connection_info()` 只给“首次安装完成”和“菜单 3 修改配置完成”使用。

### 2. 自动展示的触发条件

只有以下场景会自动展示连接信息：

- 首次安装完成并成功启动服务。
- 用户通过菜单 `3` 修改配置并成功重载服务。

以下场景不再自动展示链接或二维码：

- 已安装状态下通过菜单 `1` 仅升级 `sing-box` 二进制。
- 仅查看服务状态或其他管理操作。

对于不自动展示的路径，脚本应主动提示用户前往菜单 `8` 手动查看连接信息。

### 3. 菜单 8 的二级查看菜单

菜单 `8` 进入后先展示摘要信息，再展示二级菜单：

- `1. 仅链接`
- `2. 仅二维码`
- `3. 链接 + 二维码`
- `0. 返回`

### 4. 协议级展示规则

#### `vless+reality`

- “仅链接”展示：
  - `1. REALITY 协议链接`
  - 下一行输出完整 `vless://...`
- “仅二维码”展示：
  - `1. REALITY 协议二维码`
  - 下一行输出 ANSI 二维码
- “链接 + 二维码”同时展示以上两块。

#### `mixed`

- “仅链接”展示：
  - `1. Mixed HTTP 代理链接`
  - 下一行输出 `http://...`
  - `2. Mixed SOCKS5 代理链接`
  - 下一行输出 `socks5://...`
- “仅二维码”不展示二维码，改为明确提示：
  - `Mixed 协议当前不提供二维码，请使用链接方式手动配置客户端。`
- “链接 + 二维码”时仍只展示链接，并附带同样的提示。

### 5. 文案风格

- 先用一句话说明当前展示的内容，再输出具体值，避免直接甩出大段链接或二维码。
- 展示协议名称时必须具体，例如 `REALITY 协议链接`、`Mixed HTTP 代理链接`。
- 对 Mixed 保留 `HTTP` 与 `SOCKS5` 的明确区分。

## Architecture

本次改动保持在 `install.sh` 内完成，不引入新的脚本文件。核心做法是把当前 `display_info()` 的职责拆分成多个聚焦函数，并让不同调用点根据场景调用不同组合。

推荐的函数边界如下：

- `display_status_summary()`
  - 读取当前公共状态并展示基础摘要。
- `build_connection_artifacts()`
  - 统一组装当前协议下的链接字符串和展示文案。
- `show_link_info()`
  - 按协议输出带标题的链接信息。
- `show_qr_info()`
  - 按协议输出二维码或明确提示不支持。
- `show_connection_info_menu()`
  - 负责菜单 `8` 的二级交互。
- `show_post_config_connection_info()`
  - 在首次安装和菜单 `3` 完成后直接展示一次连接信息。

## Data Flow

### 菜单 1 首次安装

1. 生成配置并启动服务。
2. 展示基础状态摘要。
3. 展示一次连接信息：
   - `vless+reality` 显示链接 + 二维码。
   - `mixed` 显示链接，并提示无二维码。

### 菜单 1 升级二进制

1. 更新 `sing-box` 二进制。
2. 使用新版本校验现有配置。
3. 校验通过后重启服务。
4. 只展示基础状态摘要和提示语，不展示连接凭据。

### 菜单 3 修改配置

1. 加载现有配置并交互式修改参数。
2. 重写配置并重启服务。
3. 展示基础状态摘要。
4. 展示一次连接信息，视为用户显式重建配置。

### 菜单 8 查看配置信息

1. 加载配置并展示基础状态摘要。
2. 进入连接信息二级菜单。
3. 按用户选择展示仅链接、仅二维码或两者同时显示。

## Error Handling

- 若 `vless+reality` 缺少 `PUBLIC_KEY`，保留现有提示 `[密钥丢失，请更新配置]`，但仍保持新的展示结构。
- 若用户在菜单 `8` 输入无效选项，提示后重新进入该二级菜单，而不是回到主菜单。
- 若 `mixed` 选择二维码相关选项，不报错、不输出乱码二维码，只给明确提示。
- 若 `qrencode` 失败，沿用脚本当前的错误处理策略；本次不新增复杂降级逻辑。

## Testing

本次至少覆盖以下回归场景：

1. 已安装路径升级二进制时，不应自动调用连接信息展示。
2. 首次安装完成后，应调用“安装后连接信息展示”路径。
3. 菜单 `8` 在 `mixed` 协议下选择二维码时，不应调用 `qrencode`。
4. 菜单 `8` 在 `vless+reality` 下选择“仅链接”时，不应输出二维码。

测试形式以 shell 回归测试为主，沿用现有 `tests/` 目录中的临时替换脚本方式。

## Scope Boundaries

本次不做以下内容：

- 不重命名主菜单项。
- 不改动协议生成逻辑。
- 不新增图片文件或外部资源。
- 不为 Mixed 新增二维码编码格式定义。
