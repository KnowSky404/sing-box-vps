# 独立卸载脚本设计

- 日期：2026-04-21
- 适用范围：`install.sh`、新增 `uninstall.sh`、相关测试与 README
- 目标：补充一个独立的一键卸载脚本，默认执行彻底删除，并保证与菜单卸载行为一致，便于用户手动重装与实际测试

## 1. 背景

当前项目只有 `install.sh` 内部的卸载菜单项，没有独立的卸载入口。对终端用户和测试场景来说，这有两个问题：

- 用户想手动重装时，需要先进入管理菜单，不够直接
- 开发与验收场景需要频繁执行“安装 -> 卸载 -> 重装”，缺少单独脚本不利于自动化验证

## 2. 设计目标

- 新增独立 `uninstall.sh` 作为一键卸载入口
- 默认提供“彻底删除”路径，移除服务、二进制、配置目录、密钥和 `sbv`
- 菜单卸载与独立卸载必须复用同一套实际删除逻辑，避免行为漂移
- 卸载完成后不能因为日志函数副作用重新创建已删除的项目目录

## 3. 非目标

- 本轮不实现“保留配置”的真实删除分支
- 本轮不实现安装时对现存配置的探测与复用校验，只为后续工作预留范围
- 不新增额外依赖，不拆分出新的公共 shell 库

## 4. 推荐方案

采用“`install.sh` 持有核心逻辑，`uninstall.sh` 作为独立入口代理调用”的方案：

- 在 `install.sh` 中新增内部彻底卸载函数，负责真正删除服务、二进制、配置和 `sbv`
- 新增隐藏 CLI 模式，供 `uninstall.sh` 无交互调用
- `uninstall.sh` 负责模式展示、确认交互和参数透传
- 菜单卸载改为调用同一个彻底卸载函数

该方案既满足“独立脚本”，又能保持单一行为真源。

## 5. 删除范围

彻底删除模式需要覆盖：

- 停止并禁用 `sing-box` systemd 服务
- 删除 `/etc/systemd/system/sing-box.service`
- 删除 `/usr/local/bin/sing-box`
- 删除 `/usr/local/bin/sbv`
- 删除 `/root/sing-box-vps`
- 执行 `systemctl daemon-reload`

## 6. 交互策略

- `uninstall.sh` 默认进入模式选择：
  - `1. 彻底删除（默认，已实现）`
  - `2. 保留配置（预留，暂未实现）`
- 当前若选择 `2`，直接提示暂未实现并退出
- 确认执行彻底删除前再做一次 `y/N` 二次确认
- `install.sh` 菜单中的“卸载 sing-box”也保持确认提示

## 7. 验证范围

至少验证：

- 独立 `uninstall.sh` 存在并将彻底卸载请求委托给 `install.sh`
- 彻底卸载函数会删除服务文件、二进制、配置目录和 `sbv`
- 彻底卸载后不会重新创建 `/root/sing-box-vps`
- 版本元数据与 README 同步更新

## 8. 风险与控制

- 风险：删除配置目录后再调用日志函数会把目录重新建回来
  - 控制：为卸载完成路径增加不写日志文件的纯终端输出 helper
- 风险：独立脚本和菜单脚本行为分叉
  - 控制：统一由 `install.sh` 内部彻底卸载逻辑负责最终删除动作

## 9. 结论

本轮应先交付“默认彻底删除”的独立卸载脚本，并把删除逻辑统一收敛到 `install.sh`。这样既能满足用户重装与测试需求，也为下一步“保留配置”和“安装时探测旧配置”留出清晰扩展点。
