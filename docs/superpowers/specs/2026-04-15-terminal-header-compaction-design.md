# 终端菜单头部压缩与页脚信息下放设计

- 日期：2026-04-15
- 适用范围：`install.sh` 主菜单与子菜单头部区域
- 目标：压缩当前终端菜单头部高度，让主菜单更容易在单屏内完整显示，同时保留必要识别信息

## 1. 背景

上一轮终端 UI 改造已经完成菜单分区、颜色层级和摘要块统一，但在实际 VPS 终端中暴露出新的可用性问题：

- 主菜单顶部头部区域占用过高
- 作者、项目地址、版本等信息都堆在顶部，压缩了菜单主体可视高度
- 在常见终端窗口中，Logo 区和完整菜单项难以同时落在一个窗口内
- 视觉上虽然更统一，但主菜单首屏利用率不如预期

本轮只解决“头部太高”的问题，不重新设计整套菜单风格。

## 2. 设计目标

- 明显降低主菜单顶部区域的纵向占用
- 让主菜单在常见 VPS 终端高度下更接近“一屏看全”
- 保留标题、副标题和版本信息，不把识别信息删空
- 将作者和项目地址从顶部移到底部，降低其视觉权重
- 子菜单进一步压缩头部，避免每一层都重复展示低优先级信息

## 3. 非目标

- 不调整菜单编号、分区逻辑或业务行为
- 不重做主菜单和子菜单的配色系统
- 不改变当前分区标题和摘要块的整体设计方向，除非是为了减少明显多余留白
- 不承诺在所有极小终端中都绝对单屏显示全部内容

## 4. 推荐方案

采用“紧凑头部 + 主菜单页脚信息区”的方案：

- 主菜单头部保留标题、副标题、版本
- 作者和项目地址从头部移除，统一下放到主菜单页脚
- 子菜单不显示作者和项目地址
- 子菜单只保留标题，以及在必要时保留一行副标题

这样可以在不破坏当前 UI 风格的前提下，把首屏空间优先让给菜单主体。

## 5. 页面结构调整

### 5.1 主菜单

主菜单页头调整为最多 3 行有效信息：

1. 标题
2. 副标题
3. 版本

页头上下仍保留分隔线，但移除原先位于顶部的：

- 作者信息
- 项目地址
- 这些信息周围的额外垂直留白

主菜单页脚新增低优先级信息区：

- `作者: KnowSky404`
- `项目: https://github.com/KnowSky404/sing-box-vps`

该页脚位于菜单主体之后，以较低视觉权重显示，不与菜单选项竞争主视线。

### 5.2 子菜单

子菜单头部进一步收敛：

- 默认只显示标题
- 如当前页面存在明确的上下文说明需求，则最多再保留一行副标题
- 不显示版本
- 不显示作者
- 不显示项目地址

子菜单的摘要块与选项区保留现有结构，不在本轮做额外重排。

## 6. 压缩规则

- 头部上下分隔线之间不再插入多余空行
- 主菜单头部与第一组菜单分区之间仅保留必要间距
- 菜单主体和页脚之间保留 1 行缓冲，避免挤在一起
- 页脚本身只承载低权重信息，不再做额外装饰块
- 若实测仍显偏高，优先减少空白行，而不是继续删除有效信息

## 7. 信息展示策略

### 7.1 顶部保留

主菜单顶部保留：

- 标题
- 副标题
- 版本

### 7.2 底部下放

仅主菜单底部保留：

- 作者
- 项目地址

### 7.3 子菜单移除

子菜单中移除：

- 作者
- 项目地址
- 版本（默认不显示）

这样主菜单保留完整识别信息，子菜单则优先展示操作内容。

## 8. 实施边界

本次实现允许涉及：

- `install.sh` 中主菜单头部和页脚渲染逻辑
- `install.sh` 中子菜单页头信息的收敛
- 必要的菜单测试更新，用于验证头部与页脚信息位置变化

本次实现不允许涉及：

- 菜单动作和业务逻辑改写
- 配色系统全面调整
- 分区结构改写
- 新增运行时依赖

## 9. 验证范围

实现完成后至少验证：

- 主菜单头部高度较当前版本明显下降
- 主菜单顶部不再显示作者和项目地址
- 主菜单底部正确显示作者和项目地址
- 子菜单不再重复显示作者和项目地址
- 现有主菜单/子菜单菜单测试仍然通过
- 若新增测试，则验证：
  - 主菜单头部不含作者/项目地址
  - 主菜单页脚含作者/项目地址
  - 子菜单页头不含作者/项目地址

## 10. 风险与控制

- 风险：压缩过度导致主菜单识别感下降
  - 控制：保留标题、副标题和版本，不做极简到只剩一行标题
- 风险：页脚信息过弱，用户找不到项目地址
  - 控制：保留清晰文本内容，只降低位置优先级，不隐藏
- 风险：子菜单头部改动影响现有测试
  - 控制：仅删减非关键识别信息，不改动摘要块和菜单项结构

## 11. 结论

本轮应以“压缩头部高度、将作者和项目地址下放到主菜单页脚”为核心，优先提升首屏可用空间，而不是继续增强顶部展示感。目标不是更炫，而是让主菜单更适合真实 VPS 终端的操作场景。
