# HY2 随机密码与可选 OBFS 设计

## 背景

当前 `hy2` 安装流程允许使用较弱或固定的密码示例值，`obfs` 虽然已经支持，但缺少“默认安全、按需开启”的清晰交互。对于实际翻墙场景，这会带来两个问题：

- 默认密码强度不足，容易被长期复用。
- `obfs` 作为可选抗识别能力没有形成稳定工作流，开启后也缺少统一的随机密钥策略。

本设计的目标是，在不破坏现有 `hy2` 工作流的前提下，把默认安全性抬高，同时保留手动控制能力。

## 目标

- `hy2` 密码默认自动生成高强度随机值。
- 用户仍可手动输入自定义 `hy2` 密码。
- `obfs` 默认关闭。
- 用户显式开启 `obfs` 时，默认自动生成 `obfs-password`，同时允许手动输入。
- 分享链接和二维码继续基于完整 `hy2://` URI 输出；当启用 `obfs` 时，自动带上 `obfs` 与 `obfs-password` 参数。
- 维持现有 `sing-box-test` 远程验证流程，确保真实 VPS 上安装、启动、分享信息和客户端连通性都正常。

## 非目标

- 不把 `obfs` 改为默认开启。
- 不引入新的第三方依赖或密码生成工具。
- 不改变当前 `hy2` 端口、TLS、`masquerade` 等默认策略。

## 设计方案

### 1. HY2 密码生成

- 当进入 `hy2` 配置流程且用户未显式提供密码时，脚本自动生成随机密码。
- 随机值优先使用系统自带 `openssl rand`，保持与现有仓库“依赖最少、优先使用系统工具”的约定一致。
- 若 `openssl` 调用失败，则回退到现有兜底逻辑，避免安装流程被随机数生成阻塞。
- 生成结果直接写入运行态变量和协议状态文件，保证后续配置生成、节点展示、二维码输出都使用同一密码。

### 2. OBFS 交互

- `obfs` 默认保持关闭。
- 安装或重配 `hy2` 时，用户可显式选择是否启用 `salamander`。
- 若用户开启 `obfs`：
  - 默认自动生成 `obfs-password`
  - 同时提供手动输入入口
- 若用户关闭 `obfs`：
  - `SB_HY2_OBFS_ENABLED=n`
  - `SB_HY2_OBFS_TYPE` 与 `SB_HY2_OBFS_PASSWORD` 清空

### 3. 分享信息输出

- `build_hy2_link()` 继续作为单一真源生成分享 URI。
- 未开启 `obfs` 时，URI 保持：
  - `hy2://password@host:port?sni=domain#name`
- 开启 `obfs` 时，URI 自动追加：
  - `obfs=salamander`
  - `obfs-password=<generated-or-manual-password>`
- 二维码继续对完整 URI 进行编码，不引入额外格式。

### 4. 测试与验证

本次变更至少覆盖以下验证：

- 本地脚本测试
  - `hy2` 默认安装时会自动生成随机密码
  - `obfs` 默认关闭
  - 开启 `obfs` 时可自动生成 `obfs-password`
  - 手动输入密码时不会被自动覆盖
  - 分享链接在 `obfs` 开启/关闭两种状态下都正确
- 远程真实验证
  - 触发 `dev/verification/run.sh`
  - 在 `sing-box-test` 上完成 `hy2` 安装
  - 校验服务启动、配置落盘、节点展示
  - 如启用了 `obfs`，使用远程真实配置确认链接输出包含对应参数

## 兼容性与风险

- 现有未开启 `obfs` 的 `hy2` 节点不受影响。
- 已有手动密码输入能力保留，不会强制覆盖用户输入。
- 风险主要在于：
  - 随机密码生成时机不一致，导致展示信息与落盘状态不一致
  - 开启 `obfs` 后 URI 参数遗漏，导致客户端导入失败

本设计通过“单一真源变量 + `build_hy2_link()` 统一输出 + 远程真实验证”控制这些风险。
