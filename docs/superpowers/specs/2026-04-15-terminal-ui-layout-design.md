# 终端菜单 UI 居中与分区优化设计

- 日期：2026-04-15
- 适用范围：`install.sh` 菜单与菜单型摘要页面
- 目标：让主菜单与全部子菜单在当前终端窗口中尽量铺满可用宽度、整体居中、分区清晰，同时保持纯 Bash 实现且不引入额外依赖

## 1. 背景

当前脚本的菜单 UI 主要由固定宽度的 `echo` 文本组成。实际效果是：

- 顶部横幅宽度固定，无法随着终端宽度变化
- 主菜单和子菜单都采用左对齐自然流输出，在宽终端下右侧留白过多
- 菜单项是单串列表，信息分层较弱，不同功能区之间缺乏明显视觉提示
- 各子菜单的视觉风格不统一，首页之外的页面较难体现“当前所在位置”和“当前状态摘要”

本次改造只处理“显示层”，不调整业务流程、菜单动作映射、输入方式或功能边界。

## 2. 设计目标

- 终端宽度感知：根据当前终端宽度动态调整主面板宽度，而非固定字符框
- 整体居中：主菜单与全部子菜单都在可用宽度内居中展示
- 充分占宽：在安全边距内尽量拉伸 UI，减少右侧大片留白
- 分区明确：对主菜单及关键子菜单提供分组标题、色彩层级和摘要提示
- 优雅降级：窄终端、低色彩能力终端、无尺寸信息环境下仍能稳定显示
- 零新增依赖：仅使用 Bash、ANSI 转义、`tput`/环境变量，不引入 `dialog`、`whiptail` 等额外工具

## 3. 非目标

- 不改动菜单编号及其功能映射
- 不引入方向键导航、局部重绘、光标定位式伪 TUI
- 不把安装日志、实时日志、二维码等流式输出强行改造成卡片布局
- 不拆分 `install.sh` 架构，不借本轮 UI 工作进行模块化重构

## 4. 推荐方案

采用“统一 UI helper + 自适应居中面板”的方案：

- 在 `install.sh` 内新增一组专用于终端排版的 helper
- 主菜单和全部子菜单复用同一套页头、分区、菜单列表、页脚渲染能力
- 根据终端宽度决定单列或双列分区呈现
- 菜单型摘要页统一纳入新样式，非菜单型输出保留现有纵向阅读形态

该方案比“轻量增强”更能保证一致性，比“伪 TUI 重做”更符合 Bash 脚本的复杂度边界。

## 5. 布局与视觉规则

### 5.1 页面结构

所有菜单页遵循统一三段式结构：

1. 页头：标题、副标题、版本或当前上下文信息，整体居中
2. 主体：一个自适应宽度的面板，内部承载菜单分区、摘要块和选项列表
3. 页脚：输入提示、返回说明或轻量状态提示

### 5.2 宽度策略

- 使用当前终端宽度作为基础输入
- 面板宽度在“尽量铺满”和“保持安全边距”之间折中
- 宽终端下允许分区双列并排
- 中等宽度下优先单列加宽
- 窄终端下继续退化为简洁单列，避免强行双列导致换行错乱

### 5.3 颜色策略

采用“稳重科技感”风格：

- 深蓝：基础边框、主面板基调
- 青色：信息型标题、页头补充信息、说明文案
- 金色：主标题、分区标题、强调信息
- 绿色：成功状态
- 黄色：警告状态
- 红色：错误状态

颜色使用以层级表达为主，不追求高密度装饰，避免菜单沦为彩虹色块。

## 6. 组件设计

### 6.1 UI helper 范围

在 `install.sh` 中新增的 helper 只负责排版和样式，不嵌入业务逻辑。预期职责包括：

- 获取终端宽度与回退宽度
- 计算主面板宽度、左右边距和内部内容宽度
- 输出居中标题、分隔线、空白行
- 输出统一风格的分区标题
- 渲染单组菜单项
- 渲染状态摘要块
- 在适合的宽度下渲染双列分区

### 6.2 菜单项结构

菜单项统一采用：

- 编号
- 名称
- 简短说明（低一级颜色）

用户交互仍只输入数字，不增加新的输入规则。

### 6.3 摘要块结构

关键子菜单在选项列表前渲染摘要块，例如：

- 系统管理：BBR 状态
- 协议栈管理：主机网络能力、当前入站/出站协议栈、Warp 状态
- 节点信息页：当前查看模式说明或协议提示

摘要块用于帮助用户先理解当前状态，再执行菜单操作。

## 7. 菜单落地方式

### 7.1 主菜单

主菜单按功能分区组织，推荐至少拆成以下分组：

- 部署管理
- 服务控制
- 连接与诊断
- 脚本维护

保持原有编号不变，只改变视觉组织方式。

### 7.2 子菜单

以下页面统一接入新布局：

- 系统管理菜单
- 协议栈管理菜单
- 节点信息查看菜单
- 流媒检测菜单
- 其他仍然以“显示选项供用户选择”为主要目的的菜单页

子菜单统一呈现为“二级页面”风格，而不是回退到旧式左对齐文本列表。

### 7.3 非菜单输出

以下内容不纳入复杂卡片布局，只做最轻量的标题和留白统一：

- 安装/更新过程日志
- 配置校验结果
- 连接链接明细
- 二维码输出
- `journalctl -f` 实时日志

这样既保留 UI 一致性，又不牺牲可读性。

## 8. 兼容策略

### 8.1 终端宽度检测

- 优先使用可靠的终端宽度来源
- 如终端宽度不可得，回退到保守默认值
- 不依赖高级终端特性，避免在低能力终端出现错位

### 8.2 色彩能力降级

- 若终端色彩能力不足，或存在 `NO_COLOR` 环境变量，则保留布局结构，降低彩色强调密度
- 颜色不可用时，分区依然通过空白、标题、边框结构保持层次

### 8.3 清屏策略

- 菜单刷新继续使用 `clear`
- 不引入复杂的局部刷新或光标移动逻辑
- 目标是“更优雅的文本菜单”，而不是 curses 风格的全屏应用

## 9. 实施边界

本次实现允许涉及：

- `install.sh` 内新增 UI helper
- 调整主菜单与子菜单的输出结构
- 为可视化伴随产物增加 `.gitignore` 规则，避免 `.superpowers/` 污染仓库工作区
- 按项目规则递增一次 `SCRIPT_VERSION`，并同步 README 中的脚本版本展示

本次实现不允许涉及：

- 业务逻辑重写
- 协议安装流程改动
- 现有命令编号和动作映射变更
- 脚本拆模块或新建运行时依赖

## 10. 验证范围

实现完成后至少验证：

- 主菜单在常见宽度下能正常居中和占宽
- 系统管理、协议栈管理、节点信息、流媒检测等子菜单样式统一且无错位
- 窄终端下能回退到稳定单列布局
- 非菜单型输出未被新 helper 干扰
- 现有核心测试不因 UI 重排失效

## 11. 风险与控制

- 风险：终端宽度判断不稳定导致错位
  - 控制：提供默认宽度和单列回退
- 风险：颜色增强过度影响阅读
  - 控制：将颜色限定为状态与层级表达，不叠加复杂装饰
- 风险：各子菜单单独改造时风格不一致
  - 控制：所有菜单统一经过同一组 helper 渲染
- 风险：伴随界面文件落入 git 状态
  - 控制：补充 `.superpowers/` 忽略规则

## 12. 结论

本次 UI 优化应采用统一 helper 驱动的自适应居中布局，在纯 Bash 约束下实现“更饱满、更分区化、更统一”的菜单体验。实现重点是让菜单页根据终端宽度自然展开，而不是把脚本推向高复杂度的伪 TUI 架构。
